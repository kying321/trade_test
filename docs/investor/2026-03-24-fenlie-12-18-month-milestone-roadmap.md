# Fenlie 12–18 个月里程碑图文稿

> 版本：2026-03-24  
> 用途：给路演 deck、会后跟进、数据室或创始人口播提供一份统一的 12–18 个月阶段路线图。  
> 原则：**里程碑必须围绕产品化、pilot、治理与可复用性，不围绕空泛愿景。**

---

## 1. 一句话结论

Fenlie 在未来 **12–18 个月** 的核心目标，不是盲目扩功能，而是：

> **把已经形成的研究链、控制面与受控 execution bridge，从工程资产继续收口为可复用产品模板，并完成首批高质量 pilot 落地。**

---

## 2. 路线图总原则

未来 12–18 个月只围绕 4 个主目标展开：

1. **产品化收口**：把现有能力从“能用”变成“能稳定交付”
2. **pilot 落地**：拿下首批高匹配 pilot 客户并形成模板
3. **治理强化**：把 execution、acceptance、rollback、release-checklist 等治理能力做得更可对外解释
4. **模块复用**：让更多能力从高密度交付逐步过渡到可复用包结构

不建议把路线图写成：

- 追求大量品种扩张
- 追求大量页面堆叠
- 追求过早全自动 execution 商业化
- 追求不现实的收入爆炸曲线

---

## 3. 路线图结构（推荐画法）

建议在 PPT 中画成 **4 段式横向时间带**：

1. `0–3 个月`：产品化基线收口
2. `3–6 个月`：pilot 入口与交付模板建立
3. `6–12 个月`：execution bridge 产品化治理推进
4. `12–18 个月`：模块复用与更广客户验证

每一段只放：

- 2–3 个关键交付目标
- 1 个验证标准
- 1 个核心风险提醒

---

## 4. 分阶段里程碑

### Phase 1｜0–3 个月：产品化基线收口

**主目标：**

- 完成 investor/data room/pilot package 文档骨架
- 固定 Fenlie 的产品定位、成熟度边界与 execution 降温口径
- 把 Research OS、Operator Console、Evidence/Acceptance 链打磨成更稳定的演示与交付基线

**关键产出：**

1. 完整 investor pack
2. 正式版路演 deck / PDF
3. 数据室第一版
4. pilot 交付模板骨架

**验证标准：**

- 外部投资人或顾问能在不误解 execution 成熟度的前提下理解 Fenlie
- 创始人能用统一材料讲清 10–15 分钟主线

**主要风险：**

- 叙事继续漂移
- execution 被重新讲过头
- 文档很多但缺少一套统一发送逻辑

---

### Phase 2｜3–6 个月：pilot 入口建立

**主目标：**

- 明确首批目标客户画像
- 建立标准化 pilot discovery / scope / acceptance 结构
- 推动 1–2 个高质量设计合作方或 pilot 对话进入实质阶段

**关键产出：**

1. pilot customer profile 固定
2. deployment / enablement package 对外可解释
3. pilot milestone / acceptance 模板可复用
4. 首批 pilot 机会池与推进节奏

**验证标准：**

- 至少形成若干个高匹配 pipeline，而不是泛化接触名单
- pilot 讨论能够围绕明确边界，而不是“你们什么都做吗”

**主要风险：**

- 被不匹配客户拖向过度定制
- 过早承诺 execution 范围
- 没有 owner 的客户反复消耗时间

---

### Phase 3｜6–12 个月：execution bridge 产品化治理推进

**主目标：**

- 在不夸大成熟度的前提下，把 execution bridge 的治理路径做得更可复用
- 继续加强 release-checklist、acceptance、rollback、boundary note
- 把 execution 相关能力从工程路径推进到更清晰的 pilot / add-on 模块

**关键产出：**

1. execution maturity 边界持续清晰
2. execution pilot package 更标准化
3. 更完整的治理说明与验收路径
4. 与上游 Research OS / Operator Console 的衔接更稳定

**验证标准：**

- execution 不再只是“能运行”，而是“能被解释、能被验收、能被回退”
- pilot 中对 execution 的期望不再失控

**主要风险：**

- 工程实现快于治理模板沉淀
- 外部误把验证中能力当成熟产品
- 交付团队被 execution 细节拖偏主线

---

### Phase 4｜12–18 个月：模块复用与更广验证

**主目标：**

- 把高价值交付包进一步模板化
- 从单点 pilot 向更可复制的客户交付方法演进
- 验证 Fenlie 是否可以从 crypto-first 更稳地扩到 research-driven 更广机构场景

**关键产出：**

1. 更稳定的 package 化结构
2. 更清晰的 onboarding 与验收模板
3. 对更广客户类型的验证结论
4. 更完整的数据室与融资后续材料

**验证标准：**

- 至少有若干能力从“重交付”走向“可复用交付包”
- 客户价值表述更依赖系统模板，而不是临场解释

**主要风险：**

- 模板化推进慢于客户需求多样性
- 太早追求大市场而稀释 wedge
- 交付节奏与产品节奏再次脱节

---

## 5. 路演里最适合讲的 3 个核心里程碑

如果路演时间有限，建议只讲这 3 个：

1. **把控制台与研究链进一步产品化**
2. **拿下首批高匹配 pilot 客户并形成模板**
3. **把 execution bridge 从工程路径继续推进为更可复用、可治理的产品模块**

这三点已经足够解释融资用途与未来 12–18 个月的工作重心。

---

## 6. 与融资用途的映射

这份路线图应与 base round discussion 对齐：

- `$2.2M–$2.8M`
- `18–24 个月 runway`

对应关系：

### 35%｜产品与平台工程化

支撑：

- 控制台与 Research OS 产品化收口
- 架构与交付稳定性提升

### 20%｜研究链与数据增强

支撑：

- 研究治理与增强路径继续推进
- 保持 mainline 与 source-owned 纪律

### 20%｜execution bridge / 安全 / 合规治理

支撑：

- execution bridge 治理与 pilot 推进
- rollback / readiness / acceptance 边界强化

### 15%｜pilot 客户交付与成功

支撑：

- 首批高质量 pilot 落地
- 模板与 acceptance 沉淀

### 10%｜G&A / 法务 / 缓冲

支撑：

- 数据室、法务、外部推进与组织缓冲

---

## 7. 这份路线图不该承诺什么

不应承诺：

- 已锁定多少付费客户
- 已确认某个大规模收入目标
- 12 个月内一定完成成熟 execution SaaS 商业化
- 会同时覆盖所有资产类别与所有客户类型

这份路线图的任务是说明**推进逻辑**，不是制造虚假确定性。
