## 3. 推荐使用顺序

### 标准顺序

1. **先发一页版**
2. **再发市场与竞品备忘**
3. **会中用 PPT 全文案 + visual brief 出正式 deck**
4. **会后再补 BP 正文与模型**
5. **进入尽调时使用证据矩阵 + battlecard + readiness checklist 控口径**

### 不推荐顺序

- 一开始直接发 700+ 行 BP
- 未讲清“验证中”边界就先发 execution 相关内容
- 未准备证据矩阵就直接进入技术深问

---

## 4. 对外口径控制规则

所有对外材料必须遵守：

1. **已验证 / 验证中 / 规划中** 三态不混写
2. 不写收益承诺
3. 不写虚构收入、客户、ARR、估值
4. 不把 triad / orderflow 研究 sidecar 讲成主执行信号
5. 不把 execution bridge 讲成已大规模成熟 SaaS

---

## 5. 当前最稳的主叙事

当前最稳的对外主叙事是：

> Fenlie 正在构建一套面向量化团队的控制面基础设施，把研究、风控、运营控制与执行接管收敛到同一条可验证、可审计的系统链中。

不建议把主叙事讲成：

- “我们找到一个高收益策略”
- “我们马上就能做全自动执行平台”
- “我们已经是成熟商业化 SaaS”

---

## 6. 当前最适合的投资人画像

优先匹配：

- 理解量化基础设施价值的金融/量化投资人
- 能接受 crypto-first 切入口但不把终局限制在 crypto 的投资人
- 对工程纪律、系统可信度、控制面产品有认知的投资人

## 8. 新增三份辅助材料的角色

### `2026-03-24-fenlie-pitch-deck-visual-brief.md`

用途：

- 把 PPT 文案稿转成可制作的页面说明
- 指定每页该用什么图、什么版式、什么视觉重心
- 避免把 Fenlie 做成‘量化收益海报’

### `2026-03-24-fenlie-investor-email-pack.md`

用途：

- 给不同阶段的投资人沟通准备短邮件模板
- 降低首次触达与会后跟进时的口径漂移
- 明确每封邮件应附哪些材料

### `2026-03-24-fenlie-minimal-send-pack-manifest.md`

用途：

- 给创始人或顾问一个可直接执行的最小发件清单
- 固定不同阶段应附哪些文件、不应混入哪些深层材料
- 作为对外发送动作的操作面补充，避免临时拼包

### `2026-03-24-fenlie-deck-production-and-export-checklist.md`

用途：

- 把 deck full copy + visual brief 推进为正式版 PPT / PDF 成品
- 固定路演版、会前预读版、会后补发版的生产顺序
- 作为“正式成品仍待制作”阶段的制作操作清单

### `2026-03-24-fenlie-diligence-readiness-checklist.md`

用途：

- 判断哪些内容 ready / partial / not ready
- 提前拦截夸大 execution、收入、客户 traction 的风险
- 给后续数据室建设提供骨架

### `2026-03-24-fenlie-execution-maturity-note.md`

用途：

- 单独说明 execution bridge 当前的成熟度边界
- 降低在融资与尽调中对 execution 的过度承诺风险
- 帮助创始人统一回答“到底做到哪一步”

### `2026-03-24-fenlie-product-architecture-visual-brief.md`

用途：

- 生成一张可放进 deck、BP、尽调附件的产品架构单页图
- 统一解释 Research OS / source-owned control chain / Operator Console / Guarded Execution Bridge 的关系
- 防止把 Fenlie 误解成策略页或单独 execution 工具

### `2026-03-24-fenlie-data-room-index-and-request-list.md`

用途：

- 建立早期融资级数据室骨架与 request list
- 明确哪些材料 ready / partial / not ready
- 防止尽调时把不该承诺的内容硬塞进数据室

### `2026-03-24-fenlie-pilot-customer-profile-one-pager.md`

用途：

- 固定当前最值得追的 pilot 客户画像
- 统一回答“先卖给谁、为什么是他”
- 防止过早追求不匹配客户

### `2026-03-24-fenlie-deployment-enablement-package-one-pager.md`

用途：

- 解释早期交付包结构与推荐售卖顺序
- 统一说明私有部署、console 落地、enablement、execution pilot 的关系
- 把商业模式与真实交付边界对齐

### `2026-03-24-fenlie-risk-and-governance-principles-one-pager.md`

用途：

- 用一页纸收口 Fenlie 的风险治理原则
- 统一解释 source-of-truth、分层、失败分级与回退逻辑
- 让投资人与 pilot 客户都能理解 Fenlie 为什么不是黑盒自动化工具

### `2026-03-24-fenlie-pilot-milestone-and-acceptance-template.md`

用途：

- 给 pilot 项目提供标准化里程碑与验收框架
- 防止 pilot 演变为无限扩 scope 的模糊定制
- 为后续沉淀交付模板与 acceptance 标准做准备

### `2026-03-24-fenlie-12-18-month-milestone-roadmap.md`

用途：

- 统一 Fenlie 未来 12–18 个月的产品化与 pilot 路线图
- 可直接转成路演页或会后补充页
- 把融资用途与阶段目标明确对应起来

### `2026-03-24-fenlie-investor-faq-expanded.md`

用途：

- 作为 Q&A battlecard 的长版补充
- 支撑更长会谈、会后追问与数据室 FAQ 区
- 让回答更系统但仍保持边界清晰

### `2026-03-24-fenlie-unified-outreach-pack-and-notion-index.md`

用途：

- 把首次触达、会前预读、会后推进、尽调入口统一成单一发件逻辑
- 可直接迁移成 Notion 首页或共享目录索引
- 降低版本错发与口径漂移风险
