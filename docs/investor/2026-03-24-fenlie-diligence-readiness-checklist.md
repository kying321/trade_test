# Fenlie Diligence Readiness Checklist

> 版本：2026-03-24  
> 用途：用于融资推进时判断“哪些材料现在可以发、哪些只能口头解释、哪些暂时不能承诺”。  
> 原则：**宁可少说，不可错说。** 本清单不是对外宣传材料，而是内部尽调准备与口径控制文档。

---

## 1. 当前 readiness 总判断

截至 **2026-03-24**，Fenlie 已经具备一套**足以支持早期机构投资人初步评估**的材料框架，但尚未达到“成熟规模化 SaaS 尽调”所需的全部商业化成熟度。

更准确的判断是：

- **产品叙事：可对外**
- **工程证据：可对外**
- **市场与 wedge：可对外**
- **商业模型：可对外，但需强调为规划假设**
- **execution 商业成熟度：不可过度承诺**
- **客户 traction / 收入验证：若无新增事实，不可夸大**

因此目前最适合的投资人推进节奏是：

1. 先讲清系统定位与边界
2. 再证明已有工程证据
3. 再说明为什么值得投产品化收口
4. 最后才谈 execution 扩张与更远收入杠杆

---

## 2. Readiness 分级定义

### Ready

已具备文档、证据或可审计内部记录支撑，可以进入对外材料或实质问答。

### Partial

有初步依据，但容易被追问到底。可以口头讨论、谨慎书面表达，但不应作为强主张。

### Not Ready

没有足够证据支撑，或者即使有工程线索，也不适合作为对外承诺。

## 3. 尽调维度总览

| 维度 | 当前状态 | 是否可对外发送 | 主要风险 |
|---|---|---:|---|
| 项目定位与产品定义 | Ready | 是 | 需防止被误解成策略项目 |
| 工程与架构证据 | Ready | 是 | 要避免技术细节失控过深 |
| Research mainline 与治理纪律 | Ready | 是 | 不可被理解成收益承诺 |
| Operator Console / public acceptance | Ready | 是 | 截图和口径必须真实 |
| Execution bridge | Partial | 可谈，但要降温 | 容易被误读成成熟商用 execution |
| 市场 / wedge / 竞品 | Ready | 是 | TAM 需保持保守 |
| 商业模式与 3 年模型 | Partial | 是，但需标注规划假设 | 不能被当作已签收入 |
| 客户 traction / ARR / logo | Not Ready（若无新增事实） | 否 | 一旦夸大，信用直接受损 |
| 合规 / 安全 / 审计治理口径 | Partial | 可谈原则，不宜过细承诺 | 投资人可能要求更 formal 材料 |
| 财务 / 法务 / 数据室结构 | Partial | 可逐步准备 | 容易在后续 diligence 放大为短板 |

## 5. 逐项 readiness 细表

### 5.1 项目定位与问题定义

**状态：Ready**

**当前最稳口径：**

> Fenlie 是一套面向量化团队的反脆弱交易操作系统，目标是把研究、风控、运营控制与执行接管收敛到同一条可验证、可追溯、可部署的系统链中。

**已有支撑：**

- Executive Summary
- Business Plan
- Pitch Deck Full Copy
- Architecture / progress / system docs

**注意事项：**

- 不要被带偏成“我们主要卖策略”
- 不要被误解成“只是一个 dashboard”

**推荐动作：**

- 所有对外材料第一页都复用同一定位句

---

### 5.2 工程与架构可信度

**状态：Ready**

**当前可引用证据类型：**

- 架构分层
- source-owned artifact discipline
- dashboard acceptance / release-checklist
- regression / contracts / tests 数字

**已有支撑文件：**

- Investor Evidence Matrix
- Business Plan
- 系统内 architecture/progress 文档

**推荐口径：**

> Fenlie 当前不是停留在概念层，已经形成研究主线工件链、控制台公开验收链以及一组可审计的工程验证资产。

**注意事项：**

- 不要在邮件里扔过多技术术语
- 不要把“测试数”包装成 traction

---

### 5.3 Research mainline 与研究纪律

**状态：Ready**

**当前可稳讲的事实：**

- ETHUSDT / 15m / breakout-pullback 是 durable research mainline
- 当前重点是 exit/risk
- OI sidecar 已进入研究增强链
- triad 仍是 context-only

**推荐口径：**

> Fenlie 对研究输出的要求不是“有更多 signal”，而是“哪些是 canonical anchor，哪些只是 watch-only sidecar，哪些结论已经被证伪”。

**注意事项：**

- 不可将当前研究主线表述为稳定赚钱承诺
- 不可把 sidecar 误讲成执行级主信号

---

### 5.4 Operator Console / Dashboard

**状态：Ready**

**对外可讲重点：**

- public / internal 分层
- workspace / contracts / raw / alignment 穿透
- release-checklist 与 public acceptance 验收

**推荐展示方式：**

- 真实截图
- 证据矩阵引用
- 路演时作为产品实物展示

**注意事项：**

- 截图必须真实
- 不要为路演重绘一个并不存在的成熟 UI

---

### 5.5 Execution Bridge

**状态：Partial**

**当前可稳讲的口径：**

- 已完成关键工程路径与桥接治理设计
- 具备 probe / ready-check / canary / rollback 的受控接管思路
- 适合 pilot、验证与产品化推进

**不能讲的口径：**

- 已成熟商用 execution SaaS
- 已形成稳定大规模生产级客户网络
- 已在多客户场景稳定接管大规模 live capital

**推荐降温句：**

> Execution bridge 当前是验证中的产品能力，不应被对外表述为已经规模化成熟的 execution SaaS。

**投资人深问时可补：**

- 为什么要强调门禁、幂等、ready-check、rollback
- 为什么先做受控接管而不是黑盒自动化

---

### 5.6 市场、wedge 与竞品

**状态：Ready**

**当前最稳表述：**

- crypto-first 是 wedge，不是终局边界
- 先服务 crypto quant team，再扩到研究型数字资产机构技术组，再扩到更广义机构控制面

**对外可用支撑：**

- Grand View Research
- PwC + AIMA
- Coinbase Institutional + EY-Parthenon

**注意事项：**

- 外部数据必须说清是市场背景，不是 Fenlie 自有经营数据
- 竞品页重点讲空白带，不要做“全面吊打”式表述

---

### 5.7 商业模式与三年模型

**状态：Partial**

**当前可讲：**

- 早期以高价值交付为主
- 后续走向模块化授权 / 平台订阅
- 三年模型是内部规划假设
- Base round discussion 为 `$2.2M–$2.8M`

**当前不应讲得过重：**

- 明确锁定估值
- 明确锁定某轮条款
- 把 Year 2 / Year 3 模型写成高确定性收入预期

**推荐口径：**

> 当前模型主要用于说明资金用途、产品化节奏与 runway 逻辑，不构成业绩承诺。

---

### 5.8 客户 traction / logo / 收入

**状态：Not Ready（除非后续新增真实事实）**

**当前建议：**

- 没有真实、可披露、可审计事实时，不要写
- 不要为了“看起来像成熟融资项目”而补虚构 logo 或伪客户描述

**若被追问怎么办：**

可回答：

> 当前更重要的是把研究链、控制面与执行桥的产品化边界做稳。Fenlie 现阶段更适合与少量高匹配 pilot/设计合作方深度共建，而不是过早追求表面化 logo 数量。

---

### 5.9 合规 / 安全 / 风控治理

**状态：Partial**

**当前可谈：**

- 受控执行桥强调门禁、幂等、限流、rollback
- 不把通知故障、dashboard 故障与 execution ambiguity 混为一谈
- 风控不只是执行瞬间，而是研究到上线全链路纪律

**当前欠缺：**

- 更 formal 的对外合规架构说明
- 更适合放数据室的安全/治理摘要页

**建议：**

- 第一轮会议中讲原则即可
- 真进入深入 diligence 时，再整理更结构化治理说明

## 7. 投资人常问问题的“安全边界”

### 问：你们是不是已经可以大规模自动执行？

**安全回答：**

不是这样表述更准确。Fenlie 当前 execution bridge 的关键工程路径已经建立，适合 pilot 和受控验证，但我们不会把它讲成已成熟的大规模商用 execution SaaS。

### 问：你们有没有稳定赚钱的策略？

**安全回答：**

我们当前有清晰的 research mainline 和研究增强路径，但 Fenlie 的价值不在承诺单一策略收益，而在于把研究、风控、控制和执行边界做成系统资产。

### 问：你们现在收入多少？

**安全回答：**

如果目前没有可披露、可审计收入，就不要模糊回答。可以直接回到“当前阶段重心在产品化收口、pilot 设计和融资用途逻辑”。

## 9. 最后结论

Fenlie 当前已经足以支持一轮**早期机构投资人层面的认真讨论**，尤其适合那些理解量化基础设施、接受 crypto-first 楔子市场、并愿意投资“系统产品化收口”而不是短期策略故事的投资人。

> 当前最重要的不是把 Fenlie 讲得更成熟，而是把它讲得更可信。
